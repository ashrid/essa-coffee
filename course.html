The Shopping Cart

How items are tracked, stored, and persisted across browser sessions — without a database.

The Challenge

When you shop online, you expect your cart to remember what you added — even if you:

  • Close the browser tab
  • Shut down your computer
  • Come back days later

But here's the thing: Essa Cafe doesn't require customers to log in. So where does this "memory" live?

💡
The Browser as Storage

Modern browsers can store data locally using localStorage. It's perfect for shopping carts — no login required, survives browser restarts, and holds plenty of data for this use case.

How the Cart Store Works

Essa Cafe uses Zustand — a lightweight state management library — combined with a persistence middleware. Here's the magic:

CODE 
export const useCartStore = create<CartStore>()(
  persist(
    (set, get) => ({
      items: [],
      addItem: (product) => {
        const items = get().items;
        const existing = items.find(
          item => item.productId === product.productId
        );
        if (existing) {
          // Update quantity if already in cart
        } else {
          // Add new item to cart
        }
      },
    }),
    {
      name: "essa-cafe-cart",
      storage: createJSONStorage(() => localStorage),
    }
  )
);
PLAIN ENGLISH

Create a cart store that any component can use to read or update the cart.

Wrap it with "persist" middleware — this automatically saves to localStorage whenever the cart changes.

The addItem function checks if this product is already in the cart. If yes, increase quantity. If no, add it fresh.

Give the stored data a name — "essa-cafe-cart" — so the browser knows which data belongs to this app.

Use localStorage as the backend — this is what makes the cart survive browser restarts.

1

Customer clicks "Add to Cart"

→
2

Zustand updates state in memory

→
3

Persist middleware saves to localStorage

→
4

UI instantly updates to show new item

Design Decisions That Matter

✓

Client-Side Cart

Why: No login required, instant updates, works offline. Trade-off: Cart is tied to one browser/device.

🔄

Price Recalculation

Cart stores item IDs, not prices. At checkout, fresh prices are fetched from the database — preventing stale pricing bugs.

🎨

Optimistic UI

The cart drawer opens immediately on "Add" — no waiting for server. If something fails, it's handled gracefully.

04

Checkout & The Database

Where the real magic happens — validating data, preventing fraud, and ensuring orders are bulletproof.

Security Rule #1: Never Trust the Client

Here's a scenario: A sneaky customer modifies their browser's cart to change a $25 latte to $5. Then they checkout. What happens?

CODE 
// Fetch products and prices from DB
// NEVER trust client-submitted prices!
const products = await prisma.product.findMany({
  where: { id: { in: productIds } },
  select: { id: true, name: true, price: true },
});

const priceSnapshot = Object.fromEntries(
  products.map(p => [p.id, p.price])
);

// Calculate total from DB prices
const total = data.items.reduce((sum, item) => {
  return sum + Number(priceSnapshot[item.productId]) * item.quantity;
}, 0);
PLAIN ENGLISH

Look up the real prices from the database using the product IDs the customer sent.

Create a price snapshot — a lookup table of current prices, fresh from the database.

Calculate the total ourselves using database prices, completely ignoring any prices the client might have sent.

The result: Even if a customer hacks their cart, they pay the real price. The server is the source of truth.

⚠️
Common Bug to Avoid

Some e-commerce systems accept the total price from the client. This is a security vulnerability. Always recalculate on the server using authoritative data.

The Power of Transactions

What if two people try to buy the last croissant at the exact same moment? Without proper handling, you could oversell. Essa Cafe uses database transactions to prevent this.

CODE 
return prisma.$transaction(
  async (tx) => {
    // 1. Verify products are still available
    for (const item of data.items) {
      const product = await tx.product.findUniqueOrThrow({
        where: { id: item.productId },
      });
      if (!product.isAvailable) {
        throw new Error(`ITEM_UNAVAILABLE:${product.name}`);
      }
    }

    // 2. Create order + items atomically
    const order = await tx.order.create({
      data: { /* ... */ },
    });

    return order;
  },
  { isolationLevel: Prisma.TransactionIsolationLevel.Serializable }
);
PLAIN ENGLISH

Start a transaction — a protected block where all database operations are locked together.

Double-check availability — the product might have become unavailable between adding to cart and checkout.

If anything fails — throw an error and the entire transaction is rolled back. No partial orders.

Use "Serializable" isolation — the strictest level. It's like having a bouncer that lets only one transaction through at a time.

Authentication & Security

How the admin panel stays protected — without passwords that can be leaked or forgotten.

The Password Problem

Traditional passwords have issues:

  • People reuse passwords across sites (dangerous!)
  • Passwords get leaked in data breaches
  • Staff forget them and get locked out during rush hour
  • Someone has to manage password resets

Essa Cafe uses a different approach: Magic Links.

✨
Magic Links Explained

Instead of a password, you enter your email. The system sends you a special link that logs you in automatically. It's like a VIP backstage pass that's valid for 15 minutes and can only be used once. No password to remember or leak.

// Generate a secure, random token
export function generateMagicToken(email: string): string {
  const token = crypto.randomUUID();
  // Token expires in 15 minutes
  tokenStore.set(token, {
    email,
    expires: Date.now() + 15 * 60 * 1000
  });
  return token;
}
PLAIN ENGLISH

Create a random, unguessable token using the crypto module — same tech that secures bank transactions.

Store the token with the email and an expiration timestamp (15 minutes from now).

Send this token in a link to the admin's email. Only someone with email access can use it.

When clicked, verify the token hasn't expired and matches what we stored. Then log them in.

Multiple Security Layers

1
Email Whitelist

Only emails in ADMIN_EMAIL or ADMIN_EMAILS env vars can even request a magic link

2
Time-Limited

Links expire after 15 minutes — old links won't work even if intercepted

3
Single Use

Token is deleted immediately after use — can't be replayed or shared

4
Secure Transport

JWT sessions are encrypted and only transmitted over HTTPS

The QR Code Pickup System

A clever verification flow that connects the digital order to the physical pickup — no app download required.

The Pickup Problem

When a customer arrives to pick up their order, how does staff:

  • Quickly find the right order in a busy shop?
  • Verify this person actually placed the order?
  • Prevent someone from picking up the wrong order?
  • Do it all without a separate app or login?

The solution: QR codes sent via email.

CODE 
// Generate cryptographically secure token
export function generateQRToken(): {
  token: string;
  expiresAt: Date;
} {
  const token = crypto
    .randomBytes(32)
    .toString("hex");
  const expiresAt = new Date(
    Date.now() + 7 * 24 * 60 * 60 * 1000
  );
  return { token, expiresAt };
}
PLAIN ENGLISH

Generate 32 random bytes — this creates a token so long and random that guessing it would take longer than the universe has existed.

Set expiration to 7 days — QR codes are valid for a week, giving customers flexibility for pickup timing.

Store token in the order record — so we can verify it later when scanned.

Generate QR image from token — the QR code is just a visual representation of the URL containing this token.

Putting It All Together

Key takeaways, common gotchas, and how to apply these patterns to your own projects.

Patterns You Can Reuse

🛒

Client-Side Cart + Server Validation

Use localStorage for UX, but always revalidate on the server. Perfect for guest checkout flows.

🔒

Magic Link Auth

Great for admin panels or low-frequency logins. No password management overhead.

🔄

Database Transactions

Always wrap multi-step operations in transactions. Prevents inconsistent data on errors.

📧

Email as Interface

QR codes in emails bridge digital and physical worlds. No app download required.

Debugging Guide

When building with AI, these are the most common issues you'll encounter with this type of app:

🐛
"Cart shows items but checkout says empty"

Check that the cart store and checkout API use the same data shape. The API might be looking for productId while the store saves id.

🐛
"Emails not sending in production"

Gmail App Passwords are different from your regular password. Also check that GMAIL_USER and GMAIL_APP_PASSWORD are set in production environment variables.

🐛
"QR code scans but says invalid"

Check the NEXT_PUBLIC_APP_URL environment variable. If it's set to localhost, production QR codes will point to the wrong domain.

🐛
"Magic link says 'unauthorized'"

The email must be in ADMIN_EMAIL or ADMIN_EMAILS environment variable. Check for typos and case sensitivity.
