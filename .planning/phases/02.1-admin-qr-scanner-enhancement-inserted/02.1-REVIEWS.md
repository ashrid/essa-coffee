# Cross-AI Plan Review — Phase 2.1

## Gemini Review

### Summary

The implementation plan for Phase 2.1 is technically sound, well-researched, and aligns perfectly with the project's goal of providing a seamless in-browser scanning experience for admin users. By selecting `html5-qrcode` and utilizing Next.js dynamic imports (`ssr: false`), the plan correctly addresses the primary technical hurdles of browser-based camera access. The phased approach from installation to component creation and finally page integration is logical and minimizes risk.

### Strengths

*   **SSR Awareness:** The explicit use of `dynamic(..., { ssr: false })` for the `ScannerContainer` demonstrates a strong understanding of Next.js hydration requirements for browser-only APIs.
*   **Performance Optimization:** Setting `fps: 2` is a proactive measure for mobile battery conservation, which is critical for employees using the tool throughout a shift.
*   **Robust Cleanup:** The inclusion of `scanner.clear()` in the `useEffect` cleanup is essential to prevent "camera in use" errors and hardware lockups.
*   **Format Flexibility:** Supporting both raw tokens and URL-embedded tokens ensures compatibility with various QR generation methods.
*   **Fallback Strategy:** Retaining manual order entry as a fallback when camera access is denied ensures the admin remains functional in all scenarios.

### Concerns

*   **Permission API Support (MEDIUM):** The plan mentions checking `navigator.permissions.query`. Note that support for the `'camera'` permission string is inconsistent across mobile browsers (specifically Safari on iOS). Relying on this might lead to false negatives.
*   **Scan Debouncing/Double-Firing (MEDIUM):** When a QR code is detected, the `onScan` callback may fire multiple times in rapid succession. The plan should explicitly state that scanning should be "paused" or the scanner "cleared" immediately upon a successful detection to prevent multiple lookups for the same code.
*   **Facing Mode (LOW):** On devices with multiple cameras (front/back/telephoto), the scanner should explicitly request the "environment" (back) camera to avoid defaulting to the selfie camera.
*   **HTTPS for Mobile Testing (LOW):** `getUserMedia` requires a secure context (HTTPS or localhost). Testing on a physical mobile device via a local network IP (e.g., `192.168.x.x`) will fail unless a tunnel (like ngrok) or a local SSL setup is used.

### Suggestions

*   **Permission Handling:** Instead of relying on `navigator.permissions`, handle the `NotAllowedError` or `NotFoundError` caught during the scanner initialization/start phase to trigger the "permission-denied" state.
*   **Explicit Facing Mode:** In the `html5-qrcode` config, explicitly set `facingMode: "environment"` to ensure the rear camera is prioritized.
*   **Visual Feedback:** Add a brief "haptic" or visual flash/sound in the UI when a scan is successful so the user knows the code was captured before the lookup logic begins.
*   **Pause on Success:** Modify `handleScan` to call `scanner.pause()` or `scanner.clear()` immediately upon detection. This prevents the "continuous scanning" from being *too* aggressive while the admin is looking at the result of the current scan.
*   **Mobile Testing Note:** Add a note to the verification plan regarding the use of `ngrok` or similar tools to provide an HTTPS endpoint for physical mobile device testing.

### Risk Assessment: LOW

The overall risk is **LOW**. The plan uses a mature library, accounts for the most common Next.js/React pitfalls (SSR/Cleanup), and maintains a clear path back to manual entry if the technology fails. The identified concerns are minor implementation details that can be addressed during the "Act" phase of the development cycle.
