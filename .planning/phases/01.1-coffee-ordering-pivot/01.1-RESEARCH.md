# Phase 1.1: Coffee Ordering Pivot - Research

**Researched:** 2026-03-22
**Domain:** Prisma schema migration, cart/checkout flow refactoring, branding pivot
**Confidence:** HIGH

## Summary

Phase 1.1 transforms ShopSeeds into Essa Cafe by removing plant/seed product semantics and stock-quantity management. The technical pivot requires:

1. **Prisma schema changes** (drop 3 columns, add 1 boolean field) — straightforward but must be executed in the correct order to avoid breaking the running app
2. **Checkout transaction simplification** (replace stock decrement with availability check) — reduces complexity and transaction isolation requirements
3. **Cart store refactoring** (stockQuantity → isAvailable) — removes quantity-limit enforcement, simplifies component props
4. **Branding sweep** (color palette + copy replacement) — visual identity across UI, email templates, metadata
5. **Seed data replacement** (plant products → coffee menu) — 4 categories × 3-4 drinks = ~15 products

All existing patterns (atomic transactions, ISR, Zod validation, email templates) are reused. No new dependencies required.

**Primary recommendation:** Execute schema migration in 2 stages: (1) Add `isAvailable` field with @default(true), deploy, confirm app runs, (2) Remove old fields in separate migration. This prevents downtime during development and keeps the app functional if the second migration hasn't run yet.

### Pattern 1: Prisma Migration Safety (Two-Stage Deployment)

**What:** Break the schema change into two separate migrations to ensure the running app never encounters missing fields.

**When to use:** Always when dropping or renaming columns that active code reads/writes.

**Steps:**

```sql
-- Migration 1: Add isAvailable with default value (app can now read this)
ALTER TABLE products ADD COLUMN "isAvailable" BOOLEAN NOT NULL DEFAULT true;

-- Deploy app, verify it's running
-- All code now checks isAvailable instead of stockQuantity

-- Migration 2: Drop old columns (safe after code is deployed)
ALTER TABLE products
  DROP COLUMN "stockQuantity",
  DROP COLUMN "careInstructions",
  DROP COLUMN "lowStockThreshold",
  DROP INDEX "products_stockQuantity_idx";
```

**Why this matters:** If you drop columns before deploying new code, the app crashes when it tries to read them. The two-stage approach prevents downtime.

**Implementation:** Prisma will auto-generate migration files. You can customize them using `prisma migrate dev --create-only` to review before applying.

### Pattern 2: Availability Check (Replaces Stock Check)

**What:** Replace all `stockQuantity > 0` predicates with `isAvailable = true` queries.

**Before (Phase 01):**
```typescript
// Fetch featured products in stock
const products = await prisma.product.findMany({
  where: {
    isFeatured: true,
    stockQuantity: { gt: 0 },  // Stock-based availability
  },
});
```

**After (Phase 1.1):**
```typescript
// Fetch featured products available
const products = await prisma.product.findMany({
  where: {
    isFeatured: true,
    isAvailable: true,  // Boolean availability
  },
});
```

**Impact:** Simpler queries, admin control over availability (toggle per product), no decrement logic needed.

### Pattern 3: Simplified Checkout Transaction

**What:** Remove the stock-decrement loop from `createOrderAtomically`.

**Before (Phase 01 - `lib/orders.ts` lines 25-33):**
```typescript
// Step 1: Verify stock for all items
for (const item of data.items) {
  const product = await tx.product.findUniqueOrThrow({
    where: { id: item.productId },
  });
  if (product.stockQuantity < item.quantity) {
    throw new Error(`INSUFFICIENT_STOCK:${product.name}`);
  }
}
```

**After (Phase 1.1):**
```typescript
// Step 1: Verify availability (not quantity)
for (const item of data.items) {
  const product = await tx.product.findUniqueOrThrow({
    where: { id: item.productId },
  });
  if (!product.isAvailable) {
    throw new Error(`UNAVAILABLE:${product.name}`);
  }
}

// Step 4 (old code): Deduct stock atomically — REMOVED ENTIRELY
// No more stock decrement step
```

**Impact:** Simpler logic, same transaction safety (Serializable isolation still prevents race conditions for order creation itself).

### Pattern 4: Cart Store Refactoring (No Quantity Limits)

**What:** Replace `CartItem.stockQuantity` with `isAvailable` boolean.

**Before (`lib/cart-store.ts` line 11):**
```typescript
export interface CartItem {
  stockQuantity: number;  // Used to cap quantity: Math.min(qty, stockQuantity)
  quantity: number;
}

// Usage in addItem (line 38):
const newQty = Math.min(existing.quantity + 1, product.stockQuantity);
```

**After:**
```typescript
export interface CartItem {
  isAvailable: boolean;  // Just tells component if item is available
  quantity: number;
}

// Usage in addItem:
if (!product.isAvailable) {
  throw new Error("Item no longer available");
}
const newQty = existing.quantity + 1;  // No cap
```

**Impact:** Users can add unlimited quantities to cart (admin controls availability per drink, not per unit). Simplifies quantity controls in `CartItem` and `CartDrawer`.

### Anti-Patterns to Avoid

- **Don't keep `stockQuantity` for legacy reasons.** It creates confusion — is it the "max you can order" or "what's available"? Use only `isAvailable`.
- **Don't drop columns before deploying code.** Always add the new field first, deploy, confirm, then drop old fields.
- **Don't forget to update `Prisma.findMany` filters.** Search for all instances of `stockQuantity:` and replace with `isAvailable:`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Schema migrations | Custom SQL scripts | Prisma Migrate with `prisma migrate dev` | Tracks state, rollback support, type safety |
| Cart quantity limits | Custom validation in components | Rely on admin `isAvailable` toggle in DB | Avoids client-side enforcement that can be spoofed |
| Color palette | Hard-code hex values in components | Tailwind `theme.extend.colors` + CSS class names | Single source of truth, no copy-paste, easy brand updates |
| Product availability state | Client-side checks | Always read from DB at checkout | Prevents race conditions, admin control |

**Key insight:** Availability is a product attribute, not a client-side calculation. This simplification removes complexity from the cart/checkout flow and centralizes control in the admin panel.

## Common Pitfalls

### Pitfall 1: Dropping Columns Before Code Deploy

**What goes wrong:** New code expects `isAvailable`, old database still has `stockQuantity`. App throws "column not found" errors when queries run.

**Why it happens:** Excitement to clean up the database before ensuring the app can handle the new schema.

**How to avoid:** Always deploy code changes _before_ dropping columns. Use the two-stage migration pattern: (1) Add new column, (2) Deploy app, (3) Drop old columns.

**Warning signs:** Logs show `column "stockQuantity" does not exist` or `SQLSTATE 42703`.

### Pitfall 2: Missing Availability Checks in All Query Paths

**What goes wrong:** `findMany` in one route gets updated to use `isAvailable: true`, but another route still filters by `stockQuantity > 0`. Inconsistent product visibility.

**Why it happens:** Easy to miss a grep when you've got 30+ files referencing the field.

**How to avoid:** Use `grep -r "stockQuantity:" app/ lib/` before finalizing. Every match must be replaced or explicitly kept (e.g., in old test data).

**Warning signs:** Featured products show, but shop page hides them; or vice versa.

### Pitfall 3: Cart Store Not Updated to Match Schema

**What goes wrong:** Code tries to pass `stockQuantity` to cart store, but `CartItem` now expects `isAvailable`. TypeScript catches it, but runtime behavior differs (e.g., quantity cap logic breaks).

**Why it happens:** Cart store type changes are easy to overlook since they're internal.

**How to avoid:** Update `lib/cart-store.ts` first, then update all components that call `addItem()` to pass the new shape.

**Warning signs:** TypeScript errors in `ProductCard.tsx`, `AddToCartButton.tsx`, `CartItem.tsx` components.

### Pitfall 4: Forgetting `isFeatured` Checkbox in ProductForm

**What goes wrong:** Admin adds a new product, sets `isFeatured: true` in the form, but the form doesn't have a checkbox for it. Defaults to `false`, never shows up in featured section.

**Why it happens:** Easy to miss since the decision says "add isFeatured checkbox" but doesn't say "it was missing before."

**How to avoid:** Check `ProductForm.tsx` — if there's no `isFeatured` checkbox, add it. Verify by creating a test product and seeing it show on the homepage.

**Warning signs:** Admin adds a coffee drink to featured, it doesn't appear in the featured section on the homepage.

### Pitfall 5: Tailwind Color Palette Not Matching Brand Identity

**What goes wrong:** Hex values are too dark/light/warm, coffee theme doesn't look cohesive, customers think it's a plant store.

**Why it happens:** Hex values chosen without seeing them in context against the actual UI.

**How to avoid:** Test the palette values in `tailwind.config.ts` by running `npm run dev` and viewing buttons, hero section, badges. Adjust if needed.

**Warning signs:** Hero section looks gray or cold instead of warm and inviting; primary buttons don't stand out from background.

### Example 3: Availability Check in Checkout (Replaces Stock Decrement)

**Source:** Current `lib/orders.ts` (lines 25-65), modified

```typescript
export async function createOrderAtomically(
  data: CheckoutData,
  priceSnapshot: Record<string, Decimal>
) {
  return prisma.$transaction(
    async (tx) => {
      // 1. Verify availability (not stock quantity)
      for (const item of data.items) {
        const product = await tx.product.findUniqueOrThrow({
          where: { id: item.productId },
        });
        if (!product.isAvailable) {
          throw new Error(`UNAVAILABLE:${product.name}`);
        }
      }

      // 2. Calculate total (unchanged)
      const total = data.items.reduce((sum, item) => {
        return sum + Number(priceSnapshot[item.productId]) * item.quantity;
      }, 0);

      // 3. Create order + items (unchanged)
      const order = await tx.order.create({
        data: {
          guestName: data.guestName,
          guestEmail: data.guestEmail,
          guestPhone: data.guestPhone || null,
          guestNotes: data.guestNotes || null,
          paymentMethod: data.paymentMethod,
          total: new Prisma.Decimal(total),
          items: {
            create: data.items.map((item) => ({
              productId: item.productId,
              quantity: item.quantity,
              price: priceSnapshot[item.productId],
            })),
          },
        },
      });

      // 4. REMOVED: Stock decrement loop is gone
      // No concurrent decrement race conditions to worry about anymore

      return order;
    },
    { isolationLevel: Prisma.TransactionIsolationLevel.Serializable }
  );
}
```

### Example 4: Updated Cart Store Type

**Source:** Current `lib/cart-store.ts`, modified

```typescript
export interface CartItem {
  id: string;
  productId: string;
  name: string;
  price: number;
  image: string | null;
  slug: string;
  isAvailable: boolean;  // Changed from stockQuantity
  quantity: number;
}

// In addItem function:
addItem: (product) => {
  const items = get().items;
  const existing = items.find((item) => item.productId === product.productId);

  if (!product.isAvailable) {
    throw new Error("Item is no longer available");
  }

  if (existing) {
    const newQty = existing.quantity + 1;  // No cap
    const updatedItems = items.map((item) =>
      item.productId === product.productId
        ? { ...item, quantity: newQty }
        : item
    );
    set({
      items: updatedItems,
      totalItems: updatedItems.reduce((sum, i) => sum + i.quantity, 0),
      subtotal: updatedItems.reduce((sum, i) => sum + i.price * i.quantity, 0),
    });
  } else {
    const newItems = [...items, { ...product, quantity: 1 }];
    set({
      items: newItems,
      totalItems: newItems.reduce((sum, i) => sum + i.quantity, 0),
      subtotal: newItems.reduce((sum, i) => sum + i.price * i.quantity, 0),
    });
  }
},
```
